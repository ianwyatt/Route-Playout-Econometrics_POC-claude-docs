## Full SQL

```sql
CREATE MATERIALIZED VIEW cristina.oa_frames_hourly_day_of_week_index_v2 AS
WITH expanded AS (
    -- Overnight hours 0-4: distribute midnight count evenly across 5 hours
    SELECT p.original_oa_code,
           p.date_dt,
           p.day_of_week,
           gs.hr,
           p.people::numeric / 5::numeric AS people
    FROM cristina.o2_people_counts_filtered_oa_q234_with_day_of_the_week p
        CROSS JOIN generate_series(0, 4) gs(hr)
    WHERE p.hr = 0
    UNION ALL
    -- Daytime hours 5-23: use raw counts
    SELECT p.original_oa_code,
           p.date_dt,
           p.day_of_week,
           p.hr,
           p.people::numeric AS people
    FROM cristina.o2_people_counts_filtered_oa_q234_with_day_of_the_week p
    WHERE p.hr >= 5
),
people_counts_totals_by_hour AS (
    -- Aggregate people counts per OA x date x day-of-week x hour
    SELECT expanded.original_oa_code,
           expanded.date_dt,
           expanded.day_of_week,
           expanded.hr,
           sum(expanded.people) AS people
    FROM expanded
    GROUP BY expanded.original_oa_code, expanded.date_dt,
             expanded.day_of_week, expanded.hr
),
oa_hour_baseline AS (
    -- Calculate the normalisation baseline: expected footfall for each
    -- OA x day-of-week x hour, using ALL dates in the dataset
    SELECT people_counts_totals_by_hour.original_oa_code,
           people_counts_totals_by_hour.day_of_week,
           people_counts_totals_by_hour.hr,
           avg(people_counts_totals_by_hour.people) AS avg_people_oa_hr_2024,
           percentile_cont(0.5::double precision)
               WITHIN GROUP (ORDER BY (people_counts_totals_by_hour.people::double precision))
               AS median_people_oa_hr_2024
    FROM people_counts_totals_by_hour
    GROUP BY people_counts_totals_by_hour.original_oa_code,
             people_counts_totals_by_hour.day_of_week,
             people_counts_totals_by_hour.hr
)
SELECT d.original_oa_code,
       d.date_dt,
       d.day_of_week,
       d.hr,
       f.frame_id,
       d.people                                                    AS date_hour_people,
       b.avg_people_oa_hr_2024,
       b.median_people_oa_hr_2024,
       d.people / NULLIF(b.avg_people_oa_hr_2024, 0::numeric)     AS average_index,
       d.people::double precision /
           NULLIF(b.median_people_oa_hr_2024, 0::double precision) AS median_index
FROM people_counts_totals_by_hour d
    JOIN oa_hour_baseline b
        ON d.original_oa_code = b.original_oa_code
        AND d.hr = b.hr
        AND d.day_of_week = b.day_of_week
    JOIN cristina.oa_frames_map f
        ON f.oa = d.original_oa_code;
```

## Pipeline Breakdown

### Stage 1: Overnight Expansion (`expanded`)

The O2 source data has a single count at hour 0 (midnight) covering the entire 00:00–04:59 period. This CTE distributes that count evenly across 5 hours:

```
Original:  hr=0, people=500
Expanded:  hr=0, people=100
           hr=1, people=100
           hr=2, people=100
           hr=3, people=100
           hr=4, people=100
```

Hours 5–23 pass through unchanged with raw people counts.

**Implication:** Overnight index values (00:00–04:59) are based on a uniform distribution assumption, not actual per-hour measurement. This means overnight spikes at transport hubs (e.g., Lime Street, Reading station) are real in total but artificially smoothed across the 5-hour window.

### Stage 2: Aggregation (`people_counts_totals_by_hour`)

Simple `SUM(people)` grouped by OA × date × day-of-week × hour. The `SUM` handles the case where the `expanded` CTE might produce multiple rows per OA-date-hour (from the UNION ALL).

### Stage 3: Baseline Calculation (`oa_hour_baseline`)

The normalisation denominator. For each OA × day-of-week × hour, calculates:

- **`avg_people_oa_hr_2024`** — Mean across all dates in the dataset (Apr–Dec 2024). Used for `average_index`.
- **`median_people_oa_hr_2024`** — Median (50th percentile). Used for `median_index`. More robust to outliers.

**Critical design choice: day-of-week normalisation.** The baseline groups by day-of-week, not just hour. This means:

- Saturday 15:00 baseline = average of ALL Saturday 15:00 values
- Tuesday 15:00 baseline = average of ALL Tuesday 15:00 values

A Saturday football match is compared against typical Saturdays, not typical Tuesdays. This makes the index more conservative — a match-day spike has to exceed the average Saturday, which already includes other match days.

**This is by design and consistent with how Route models audience.** Route models typical behaviour — if a venue hosts events most Saturdays, the Saturday baseline reflects event-level footfall, and Route's base audiences should already capture that. The MI correctly shows modest deviation at high-frequency venues (Wembley max 1.62x) and large deviation at infrequent-event venues (Sunderland 35x for a one-off Springsteen concert). The MI adds the most value where events are atypical for that location.

### Stage 4: Index Calculation (final SELECT)

```
average_index = actual_people / avg_baseline
median_index  = actual_people / median_baseline
```

- Index = 1.0 means exactly average for that OA at that day-of-week and hour
- Index > 1.0 means above-average footfall
- Index < 1.0 means below-average footfall

The `NULLIF` prevents division by zero for OAs with no baseline data.

### Stage 5: Frame Mapping

The join to `cristina.oa_frames_map` maps Output Areas to frame IDs. Multiple frames in the same OA share identical index values. This is the OA-level granularity limitation documented elsewhere.

## Key Properties

| Property | Value |
|----------|-------|
| **Granularity** | Output Area × date × hour |
| **Normalisation** | Per OA × day-of-week × hour |
| **Baseline period** | All dates in dataset (Apr–Dec 2024) |
| **Mean index** | Approaches 1.0 across all frame-hours (by construction) |
| **Overnight model** | Uniform distribution of midnight count across hours 0–4 |
| **Frame mapping** | OA → frame via `oa_frames_map` (many frames per OA) |

## Known Implications

1. **Day-of-week normalisation is by design**: The MI measures deviation from typical day-of-week behaviour. At frequently-busy venues, events are typical — the MI correctly shows modest deviation because Route's base audiences should already capture that footfall. At infrequent-event venues, the MI shows large deviation. This is the MI working as designed, not a flaw.

2. **OA sharing**: Old Trafford and Trafford Centre, or any co-located venues/retail in the same OA, produce indistinguishable index values.

3. **Overnight smoothing**: The 5-way split of midnight counts is a modelling choice. Real overnight activity may be concentrated in specific hours (e.g., 23:00–01:00 post-event egress) but appears uniform in the index.

4. **BST→GMT artefact**: On 27 Oct 2024, the clock change means hours 01:00–04:00 may have duplicated or anomalous values in the source data, which then flow into the index.

5. **Event days in the baseline**: The baseline includes event days. A venue with 10 event days out of 39 Saturdays has its Saturday baseline reflecting event-level footfall. The MI correctly measures deviation above what is typical, not absolute uplift. If Route's audience models do not fully capture regular event footfall at high-frequency stadiums, that is a limitation of Route's data inputs, not the MI methodology.

6. **Stadium OA coverage gap**: All four major stadiums audited (Wembley, Emirates, Stamford Bridge, Etihad) have OOH frames within 200m but none carry MI data — stadiums lack the retail/supermarket formats that dominate MI coverage. Event impact is only visible through spillover into adjacent commercial OAs.
